# FileShare 要件定義書

最終更新: 2026-06-26

## 1. 概要

WordPress 上でセキュアなファイル共有を実現するプラグイン「**FileShare**」。
アップロードしたファイルを「ダウンロード ID + パスワード」で保護し、回数制限・期限切れ自動削除・不正アクセス対策を備える。

## 2. 動作環境

| 項目 | 要件 |
|------|------|
| WordPress | 6.2 以上 |
| PHP | 8.1 以上 |
| PHP 拡張: sqlite3 | 必須（ログ機能・IP制限・レート制限で使用） |
| PHP 拡張: zip | 必須（複数ファイルの自動 ZIP 圧縮で使用） |

> ※ 有効化時に上記要件を自動チェックし、満たさない場合は有効化を中断する。

## 3. 機能要件

| No. | 機能 | 内容 | 実装状況 |
|-----|------|------|----------|
| F-1 | ファイルアップロード | ドラッグ&ドロップ対応。複数ファイルは自動 ZIP 圧縮 | 実装済 |
| F-2 | ファイル管理 | 保存ファイルの一覧表示・削除（アップロード画面） | 実装済 |
| F-3 | 安全なダウンロード | ダウンロード ID + パスワードによる認証。固定ページ不要の専用URL（`?fileshare_download=1&fsid=XXXX`）で認証フォームを表示し、アップロード結果にその URL を提示 | 実装済 |
| F-4 | ダウンロード回数制限 | ファイルごとに最大ダウンロード回数を設定可能（0=無制限） | 実装済 |
| F-5 | 自動クリーンアップ | 期限切れ・回数超過ファイルをバックグラウンドで自動削除 | 実装済 |
| F-6 | IP ブルートフォース対策 | 連続ログイン失敗による IP ブロック | 実装済 |
| F-7 | レート制限 | エンドポイント単位のアクセス制限 | 実装済 |
| F-8 | ダークテーマ UI | GitHub ライクな見やすいインターフェース | 実装済 |
| F-9 | フロントアップロード | 固定ページ不要の専用URL（`?fileshare_upload=1`）でアップロード画面を表示。ダッシュボードで設定した専用 ID/パスワード（1 組）でログインしてアップロード。WordPress ユーザー認証とは独立 | 実装済 |

### 補足: レート制限（F-7）について

元要件では「flask-limiter によるエンドポイント単位のアクセス制限」と記載されていたが、
flask-limiter は Python/Flask 専用ライブラリであり WordPress（PHP）では利用できない。
そのため**同等の機能を PHP + SQLite で自前実装**している（`FileShare_Security::is_rate_limited()`）。

- 既定値: ダウンロード認証エンドポイントで **1 分あたり 10 回** まで。

## 4. 非機能要件・セキュリティ要件

| No. | 項目 | 内容 |
|-----|------|------|
| N-1 | ファイル保護 | 実ファイルは `wp-content/uploads/fileshare-files/` にランダム名で保存し、`.htaccess` で直接アクセスを遮断。配信は PHP 経由のみ |
| N-2 | パスワード保管 | `password_hash()`（bcrypt）でハッシュ化保存 |
| N-3 | ダウンロード認証 | 認証成功時のみ短命（既定 600 秒）の HMAC 署名トークンを発行 |
| N-4 | IP ブロック | 既定: 15 分以内に 5 回失敗で 30 分ブロック |
| N-5 | 入力検証 | nonce 検証、権限チェック（`upload_files`）、`$wpdb` プリペアドステートメント、アップロード MIME/拡張子チェック |
| N-6 | ログ記録 | アクセス・ログイン試行・ブロック・レート制限を SQLite に記録（既定 30 日保持） |
| N-7 | アンインストール | 削除時にテーブル・保存ファイル・SQLite・オプションを完全除去 |
| N-8 | フロントログイン認証 | 専用 ID/パスワードは `password_hash()` で保存。認証成功時に HMAC 署名付きセッショントークンを発行し、HttpOnly / SameSite=Lax Cookie（既定 1 時間）で保持。ログイン試行は IP ブロック・レート制限の対象。ユーザー名変更で既存セッション失効 |

> ※ Nginx 環境では `.htaccess` が無効のため、`location ~ /uploads/fileshare-files/ { deny all; }` 相当の設定を別途必要とする（運用上の注意）。

## 5. 権限

| 操作 | 必要権限 |
|------|----------|
| アップロード / ファイル管理 / 削除（管理画面） | `upload_files` |
| アクセスログ閲覧 | `manage_options` |
| フロント設定（専用 ID/パスワードの設定） | `manage_options` |
| ダウンロード（フロント） | 不要（ID + パスワード認証のみ） |
| アップロード（フロント専用URL `?fileshare_upload=1`） | WordPress 権限は不要（専用 ID + パスワード認証のみ） |

## 6. データ保存

| 種別 | 保存先 |
|------|--------|
| ファイルメタデータ | MySQL カスタムテーブル `{prefix}fileshare_files` |
| ログ・IP制限・レート制限 | SQLite `wp-content/uploads/fileshare-files/fileshare-logs.sqlite` |
| ファイル本体 | `wp-content/uploads/fileshare-files/`（ランダム名） |
| フロントアップロード設定（ID・パスワードハッシュ・有効フラグ） | WordPress オプション `fileshare_front_upload` |

## 7. テスト

- 実行環境: PHP 8.3.31（winget 導入）+ PHPUnit 11.5.55（`tools/phpunit.phar`）。
- 方式: WordPress 非依存の単体テスト（WP 関数はスタブ化、SQLite は実 DB を一時ディレクトリで使用）。
- 実行: `pwsh tools\run-tests.ps1`
- テスト対象:
  - 認証トークン（HMAC）の発行・検証・改ざん検出
  - フロントログインのセッショントークン（HMAC）の発行・検証・期限切れ・改ざん・ユーザー名差し替え検出
  - アップロードの `$_FILES` 正規化
  - SQLite ログ／IP ブロック／レート制限
  - クライアント IP 取得・ブルートフォースブロック
  - フロントアップロード専用URL（`?fileshare_upload=1`）の生成
- 最新結果: **OK (25 tests, 40 assertions)** 全件成功。
